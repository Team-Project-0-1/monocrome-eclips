# 문서 기반 구현 출처 대장

이 파일은 게임 콘텐츠를 구현할 때 어떤 내용이 Google Drive 문서에서 확정되었고, 어떤 내용이 현재 코드 구조 때문에 임시 적용되었는지 추적하기 위한 내부 문서입니다.

## 원칙

- Google Drive의 `Project 0.1% / 03. 동전 전투 (포폴)` 문서를 콘텐츠 기준으로 삼는다.
- 문서에 없는 몬스터, 스테이지, 스킬, 이벤트, 보상은 새로 만들지 않는다.
- 코드 구조상 임시로 맞춘 항목은 반드시 `추후 대체/변경 필요`로 남긴다.
- 개발 모드에서는 `validateContentManifest()`로 플레이 가능 스테이지의 몬스터/패턴 연결 누락을 콘솔에 표시한다.

## 스테이지 반영 상태

| 스테이지 | 문서 상태 | 코드 상태 |
| --- | --- | --- |
| Stage 1: 벙커 외곽 | 전투 그룹과 기본 루프 구현 가능 | 기존 빌드 유지 |
| Stage 2: 중립 지대 | `몬스터 컨셉` 문서에 일반 몬스터 3종, 미니보스 1종, 보스 1종 상세 기재 | 문서 수치 기반으로 몬스터/패턴 추가 |
| Stage 3: 최종 심층 | `기획서` 기준 관문 루프, 3관문 적 수/필드 최대치, 보스 후 `비기` 보상 구조만 확인됨 | 구조 문서만 추가하고 플레이 불가 잠금 처리. 몬스터/보스/이벤트 상세는 추후 대체/변경 필요 |

## 패시브 구현 반영

상세 구현 정의와 임시 판정은 [passive-implementation-map.md](./passive-implementation-map.md)에 정리했다.
신버전 캐릭터 시트 기준으로 `dataUpgrades.ts`에 존재하던 60개 패시브는 보상/전투 엔진 참조 경로까지 연결되었다.

아래 항목은 코드 구조 또는 문서 수치 충돌 때문에 임시 판정을 포함한다. 추후 대체/변경 필요.

- `TANK_P_ABSORB_DEFENSE`: 방어가 실제 차감되지 않는 현재 모델에서 유효 방어 초과 HP 피해를 방어 파괴로 판정.
- `ROGUE_P_DUAL_WIELD`: 같은 턴에 스킬 임시 추적 2배 효과가 있으면 스킬 임시효과를 우선.
- `MAGE_P_RESONANCE_SHIELD`: 흡수 10% 증가는 `ceil(공명 * 0.1)`로 계산.
